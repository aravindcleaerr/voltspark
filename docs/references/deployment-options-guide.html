1. Key Concepts & Terminology

1.1 Hosting Terminology

Bandwidth
Total data transferred from server to users per month. Every page load (HTML, CSS, JS, images) consumes bandwidth. A typical page = 1-3 MB. So 100 GB ≈ 30,000–100,000 page loads/month. When exhausted, your site stops serving or gets throttled.
Compute / Invocations
Server-side processing capacity. Each time a user loads a page with server-side logic (API call, database query, SSR), that's one "invocation." 150K/month ≈ 5,000 requests/day. This is the brain of your app — when exhausted, your app's dynamic features stop working.
Build Minutes
Time spent compiling/building your app during deployment. Each git push triggers a build (typically 1-5 minutes for Next.js). 6,000 min/month = ~1,200 deployments. Rarely a limiting factor unless you push very frequently.
Cold Start
When no one visits for a while, serverless platforms "sleep" your code. The next visitor must wait while the platform "wakes up" your app. 0s = always ready. 1-3s = noticeable but acceptable. 25-30s = terrible UX, visitors will leave. This is often the #1 UX differentiator between platforms.
Edge Network
A global network of servers (CDN) that serves your content from the nearest location to the user. Cloudflare has 300+ locations, Vercel has 70+. Edge deployment means your app runs close to users worldwide, reducing latency.
Serverless
Your code runs in short-lived containers that spin up on demand and shut down after use. You pay only for actual usage, not idle time. Trade-off: cold starts. Most modern platforms (Vercel, Cloudflare, AWS Lambda) use this model.
SSR (Server-Side Rendering)
HTML is generated on the server for each request. Requires compute resources. Next.js Server Components use this approach. Contrast with SSG (Static Site Generation) where HTML is pre-built at deploy time.
ISR (Incremental Static Regeneration)
A Next.js feature that rebuilds static pages in the background at configurable intervals. Combines the speed of static pages with the freshness of server rendering. Not all platforms support this equally.

1.2 Database Terminology

SQLite
A lightweight, file-based database that stores everything in a single .db file. No separate server needed. Perfect for development and small apps. Limitation: doesn't work on serverless platforms (no persistent filesystem). Requires a cloud-hosted version (Turso, D1) for deployment.
PostgreSQL
The industry-standard open-source relational database. Powerful, feature-rich, with strong data integrity. Runs as a separate server. Supported by all major cloud platforms. The safest long-term choice for production apps.
ORM (Prisma)
Object-Relational Mapping — a tool that lets you write database queries in TypeScript instead of raw SQL. Prisma generates type-safe queries, handles migrations, and supports multiple databases. Changing databases requires updating the Prisma provider and potentially adjusting the schema.
Connection Pooling
In serverless environments, each function invocation creates a new database connection. Too many connections can overwhelm the database. Connection pooling (e.g., Prisma Accelerate, PgBouncer, Neon Pooler) manages a shared pool of connections to prevent this.
Database Adapter
A package that bridges Prisma with a specific database driver. Examples: @prisma/adapter-libsql (for Turso), @prisma/adapter-neon (for Neon serverless), @prisma/adapter-d1 (for Cloudflare D1). The adapter handles protocol differences between Prisma and the database.
Cold Start (Database)
Same as hosting cold starts, but for the database. Some free-tier databases "sleep" after inactivity. Turso: never sleeps. Neon: sleeps after 5 min (1-3s wake). Supabase: sleeps after 7 days (requires manual wake-up click).
Migrations
Schema changes (adding tables, columns, etc.) tracked as versioned SQL files. prisma migrate dev generates migration files. Some cloud databases (Turso, D1) don't support Prisma Migrate directly — you use prisma migrate diff instead.

2. Hosting Platforms Comparison

2.1 Column Definitions

Platform
The hosting service name and its deployment model.
Bandwidth
Monthly data transfer limit. How much content can be served to visitors. 100 GB is generous for most apps. "Unlimited" (Cloudflare) means no cap, but fair-use policies apply.
Compute
Server-side processing allowance. Measured in invocations (function calls), requests, or compute hours depending on the platform. This powers your API routes, database queries, and SSR pages.
Build Minutes
Minutes available per month for compiling your app during deployments. A Next.js build typically takes 1-5 minutes. 6,000 min = plenty of room for frequent deploys.
Cold Start
Time for the first request after the app has been idle. Lower is better. 0s = always warm. >10s = unacceptable for most users.
Framework Support
How well the platform handles your web framework (Next.js, React, Vue, etc.). "First-class" means native, optimized support. "Via adapter" means it works through a third-party bridge.
Credit Card
Whether a credit card is required to sign up for the free tier. "No" is preferred — zero risk of unexpected charges.
Commercial Use
Whether the free tier allows business/client projects. Vercel Hobby is personal-only. Cloudflare allows commercial use on free tier.
Permanent Free
Whether the free tier lasts forever or is a time-limited trial. "Yes" = free indefinitely within limits. "No" = trial that expires.
Score
Weighted overall rating (out of 10) considering: generous limits, cold start performance, framework support, ease of use, no credit card requirement, and long-term viability.

2.3 Detailed Platform Profiles

Vercel — Best for Next.js

Built by the same team behind Next.js. Auto-deploys from GitHub on every push. Fluid compute eliminates cold starts for 99.37% of requests. Hobby plan is free forever but restricted to personal/non-commercial use. For client projects, Pro plan is $20/mo.

Best for: Next.js apps, personal projects, demos, prototypes. The default choice unless you have a specific reason not to use it.

Cloudflare Pages/Workers — Best for Global Performance

Runs on 300+ edge locations worldwide. Unlimited bandwidth on free tier. Uses OpenNext adapter for Next.js (not native but works well). Also offers D1 (SQLite database), KV (key-value store), and R2 (object storage) — all with free tiers. Commercial use allowed on free tier.

Best for: Globally distributed apps, bandwidth-heavy sites, commercial projects that need free hosting. The most generous free tier overall.

Netlify — Good Alternative to Vercel

Similar to Vercel but not built by the Next.js team. Legacy accounts (pre-Sep 2025) have generous limits. New accounts use a credit-based system that is significantly less generous. Good for simpler sites.

Watch out: Post-Sep 2025 accounts get a much smaller free tier. Check your account creation date.

Render — Simple but Slow

Runs your app as a standard Node.js service. Easy to deploy. But free services sleep after 15 minutes and take 25-30 seconds to wake up. PostgreSQL database expires after 30 days.

Dealbreaker: 25-30 second cold starts make it unusable for any app where users expect fast response times.

AWS Amplify — Powerful but Complex

Full AWS ecosystem access. Excellent for enterprise-grade apps. But requires credit card, has complex billing, and only 1 GB bandwidth on free tier. Cold starts of 6-8 seconds on Lambda.

Best for: Teams already invested in AWS. Overkill for small-to-medium projects.

Railway & Fly.io — No Free Tier

Both are excellent paid platforms but offer only short trials (30 days / 7 days). Not viable for free deployment. Consider Railway ($5/mo) or Fly.io (pay-as-you-go) when you outgrow free tiers.

3. Database Options Comparison

3.1 Column Definitions

Database
The cloud database service name.
Type
The database engine. SQLite/libSQL = lightweight, file-based (what our apps often start with). PostgreSQL = industry-standard relational DB. MySQL = another popular relational DB. MongoDB = document/NoSQL database (different data model).
Free Storage
How much data you can store for free. A typical small-to-medium app with thousands of records uses 5-50 MB. Even 500 MB is generous for most use cases. Only matters for apps with large datasets (images, logs, analytics).
Prisma Support
How well the database integrates with Prisma ORM. Excellent = officially supported, stable, documented. Good (adapter) = works via adapter package. Preview = experimental, may have bugs or missing features like transactions.
Cold Start
Whether the database goes to sleep when idle. None = always responsive. 1-3s = brief pause after idle period. 7-day pause = requires manual action to wake up (Supabase). Affects user experience on first visit after idle time.
Credit Card
Whether a credit card is needed for the free tier. "No" = zero financial risk.
Migration Effort
How much code change is needed to switch from local SQLite. Minimal = change adapter + connection string only (Turso). Medium = change Prisma provider, adjust schema types, re-run migrations (PostgreSQL). High = redesign data model (MongoDB).
Permanent Free
Whether the free tier lasts forever. Some databases expire (Render: 30 days) or have been discontinued (PlanetScale). "Yes" = free indefinitely within limits.
Score
Weighted overall rating considering: free tier generosity, Prisma support, cold start behavior, migration simplicity, and long-term reliability.

3.3 Detailed Database Profiles

Turso — Best for SQLite Apps

Cloud-hosted libSQL (SQLite fork). Your app keeps the sqlite Prisma provider — just swap the adapter from better-sqlite3 to libsql. No schema changes, no type adjustments, no migration headaches. 5 GB storage and 500M reads/month is extremely generous. Zero cold starts since March 2025.

Best for: Apps already using SQLite + Prisma. The easiest migration path from local development to cloud deployment.

Neon — Best PostgreSQL Option

Serverless PostgreSQL with auto-scaling. Officially endorsed by Vercel (Vercel Postgres redirects to Neon). Excellent Prisma support with dedicated serverless driver. 0.5 GB per project, 100 CU-hours/month. Database branches for development workflows.

Best for: Apps that need PostgreSQL features (complex queries, JSONB, full-text search) or plan to scale significantly. The industry-standard choice.

Supabase — All-in-One Platform

PostgreSQL database + Auth + Storage + Edge Functions in one platform. Great value if you use multiple features. But free-tier projects auto-pause after 7 days of inactivity, requiring manual unpause — unacceptable for any production app.

Watch out: The 7-day auto-pause makes Supabase free tier unsuitable for apps that aren't visited daily. Fine for active development, risky for demos or low-traffic apps.

Cloudflare D1 — Edge-Native SQLite

SQLite running on Cloudflare's edge network. Fast globally. But only works with Cloudflare Workers (can't use with Vercel or other hosts). Prisma adapter is in Preview — no transaction support, and Prisma Migrate doesn't work directly.

Best for: Apps fully committed to the Cloudflare ecosystem. Not suitable as a standalone database for other hosting platforms.

5. Decision Tree — How to Choose

Q1: Is your app built with Next.js?
Yes → Use Vercel for hosting (best Next.js support)
No (React SPA, Vue, Svelte, etc.) → Consider Cloudflare Pages or Netlify

Q2: Does your app currently use SQLite?
Yes → Use Turso (minimal migration, keep sqlite provider)
No, or willing to switch → Go to Q3

Q3: Do you need advanced database features (JSONB, full-text search, complex joins)?
Yes → Use Neon (PostgreSQL)
No → Turso is simpler and has zero cold starts

Q4: Is this a commercial / client project?
Yes → Vercel Pro ($20/mo) or use Cloudflare (free commercial use)
No (personal, demo, internal) → Vercel Hobby (free) is fine

Q5: Do you need global edge performance?
Yes → Cloudflare Workers + Turso (both edge-native)
No → Vercel is sufficient (US/EU regions)

Q6: Do you want zero credit card risk?
Yes → Vercel + Turso or Vercel + Neon (no card required)
Don't mind → CockroachDB or AWS options also available

7. Migration Effort Guide

What's involved in switching from local SQLite to each cloud database:

Minimal Migration — Turso (libSQL)

Estimated time: 30 minutes

# 1. Install libsql client
npm install @libsql/client

# 2. Update prisma.ts - change adapter
# FROM: import { PrismaBetterSqlite3 } from '@prisma/adapter-better-sqlite3'
# TO:   import { PrismaLibSQL } from '@prisma/adapter-libsql'
#       import { createClient } from '@libsql/client'

# 3. Update connection
# const client = createClient({ url: process.env.TURSO_DATABASE_URL, authToken: process.env.TURSO_AUTH_TOKEN })
# const adapter = new PrismaLibSQL(client)

# 4. schema.prisma stays the same (provider = "sqlite")

# 5. Push schema to Turso
npx prisma db push

# 6. Done!

Medium Migration — Neon / Supabase / PostgreSQL

Estimated time: 2-3 hours

# 1. Update schema.prisma
# Change: provider = "sqlite"  →  provider = "postgresql"

# 2. Adjust schema types (common changes):
# - Boolean fields: SQLite uses 0/1, PostgreSQL uses true/false (Prisma handles this)
# - DateTime: May need @db.Timestamp annotations
# - String length: Add @db.Text for long text fields
# - Auto-increment: SQLite AUTOINCREMENT vs PostgreSQL SERIAL

# 3. Install PostgreSQL adapter
npm install @prisma/adapter-neon @neondatabase/serverless

# 4. Generate fresh migration
npx prisma migrate dev --name init

# 5. Update prisma.ts with new adapter

# 6. Re-run seed script

# 7. Test all queries (some SQLite-specific queries may need adjustment)

High Migration — MongoDB

Estimated time: 1-2 days. Not recommended unless you have specific NoSQL requirements.

# Requires complete data model redesign:
# - Relational joins become embedded documents or references
# - Schema changes from @relation to manual references
# - Many Prisma features unavailable (cascade deletes, etc.)
# - Different query patterns throughout the app
# NOT RECOMMENDED for apps built with relational data models

Key Security Practices for Deployment:
  • Never commit .env files to git (use .gitignore)
  • Use platform environment variables for secrets (API keys, database URLs, auth tokens)
  • Enable 2FA on all platform accounts (GitHub, Vercel, Turso, etc.)

  • Use connection pooling to prevent connection exhaustion attacks
  • Set up monitoring and alerts for unusual traffic patterns

10. Deployment Checklist

Use this checklist for every new project deployment:

Pre-Deployment

  • Code pushed to GitHub (private repo recommended)
  • .gitignore excludes: node_modules, .env, *.db, generated files
  • No secrets/API keys committed to the repo
  • App builds successfully locally (npm run build)
  • All tests pass
  • Database schema is finalized

Platform Setup

  • Create hosting platform account (Vercel/Cloudflare/etc.)
  • Create database account (Turso/Neon/etc.)
  • Create database and note the connection URL + auth token
  • Connect GitHub repo to hosting platform
  • Set root directory (if app is in a subdirectory, e.g., app/)
  • Configure build command and output directory

Environment Variables

  • Set DATABASE_URL (or TURSO_DATABASE_URL)
  • Set TURSO_AUTH_TOKEN (if using Turso)
  • Set NEXTAUTH_SECRET (generate: openssl rand -base64 32)
  • Set NEXTAUTH_URL to production domain
  • Set any other app-specific environment variables

Database Setup

  • Push schema to cloud database (npx prisma db push)
  • Run seed script if needed
  • Verify data in cloud database

Post-Deployment

  • Verify app loads at production URL
  • Test login/authentication
  • Test all CRUD operations
  • Check for console errors in browser
  • Test on mobile devices
  • Set up custom domain (optional)
  • Enable HTTPS (usually automatic)
  • Set up monitoring/alerts (optional)

11. Glossary

Term Definition
API RouteA server-side endpoint in your Next.js app that handles HTTP requests (GET, POST, PUT, DELETE). Lives in app/api/ directory.
CDNContent Delivery Network. A global network of servers that caches and serves your static assets (images, CSS, JS) from the nearest location to the user.
CI/CDContinuous Integration / Continuous Deployment. Automated pipeline that builds, tests, and deploys your code when you push to GitHub.
CU (Compute Unit)Neon's unit of compute. 1 CU = 1 vCPU + 4 GB RAM. Free tier gets 100 CU-hours/month (enough for light usage).
EdgeServers distributed globally (edge of the network, close to users). Edge functions run on these servers for low latency.
EgressData leaving the server/database. Bandwidth charges are typically based on egress (outbound data).
InvocationOne execution of a serverless function. Each API request, SSR page render, or server action = 1 invocation.
ISRIncremental Static Regeneration. A Next.js feature that regenerates static pages in the background at set intervals.
libSQLAn open-source fork of SQLite created by Turso. Adds features like HTTP access, replication, and cloud hosting while maintaining SQLite compatibility.
ORMObject-Relational Mapping. A tool (like Prisma) that lets you interact with databases using programming language objects instead of raw SQL.
Provider (Prisma)The database type in your Prisma schema: sqlite, postgresql, mysql, mongodb, cockroachdb.
Request Unit (RU)CockroachDB's billing unit. Each database operation consumes a certain number of RUs based on complexity.
Scale to ZeroA feature where your database/compute automatically shuts down when not in use, costing nothing. Trade-off: cold starts when traffic returns.
ServerlessCode runs in ephemeral containers that start on demand and shut down after use. No persistent server to manage. Pay only for usage.
SSGStatic Site Generation. HTML pages are pre-built at deploy time. Fastest serving method but content is static until next deploy.
SSRServer-Side Rendering. HTML is generated on the server for each request. Allows dynamic, personalized content but requires compute resources.
TTFBTime To First Byte. How long it takes for the server to send the first byte of response. Directly affected by cold starts.
